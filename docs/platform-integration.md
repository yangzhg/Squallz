# Squallz Platform File-Manager Integration

Updated: 2026-06-12

This document defines the non-macOS integration boundary. macOS
Finder Quick Actions are implemented in the Tauri app. Windows and Linux
integration artifacts are generated as reviewable/manual packs until those
platforms are restored as active tested targets.

## Product Rules

- Do not silently take default archive ownership.
- Do not install shell actions without explicit user action.
- Context actions must call the shared `sqz` CLI, not duplicate archive logic.
- Archive business behavior must remain in core/formats/recovery and CLI.
- Platform shell integration is allowed to be platform-specific, but it must be
  reversible and documented.

## macOS

Finder Quick Actions are installed by the packaged Tauri app and are exercised
through `/usr/bin/automator` by `scripts/macos_packaged_quick_actions_smoke.sh`.
That proves the generated workflows can resolve the packaged first-party
`Contents/Resources/bin/sqz` helper and run real archive operations.

Visible Finder menu validation has a separate permission layer. Run
`scripts/macos_finder_ui_preflight.sh` before an attended Finder right-click or
physical drag session; it checks Finder AppleEvents, System Events Accessibility
UI scripting, and Finder UI process inspection. It does not click the Finder
context menu and does not replace the final visible Finder UI sign-off.

The final visible Finder context-menu layer is exercised by
`scripts/macos_finder_context_menu_smoke.sh`. That script temporarily installs a
single-purpose `Squallz Finder Context Smoke.workflow` into the real
user-visible `~/Library/Services`, reveals a fixture ZIP in Finder, clicks the
Finder context menu action through System Events, and verifies the selected file
was tested by the packaged `Contents/Resources/bin/sqz` helper. If the current
automation Space exposes no visible Finder window after reveal, the script must
write a blocked report instead of passing; rerun it from an attended visible
desktop session for final sign-off.

## Windows

The Tauri app can install classic per-user Explorer verbs under
`HKCU\Software\Classes` and user-local PowerShell wrapper scripts under the
Squallz data directory. Each verb first launches the shared GUI task-window
handoff (`--squallz-action`, plus `--squallz-output` when needed) and only falls
back to the shared `sqz` CLI if the GUI handoff is unavailable.

This is intentionally not marked complete for Win11: modern Win11 top-level
context menus require `IExplorerCommand` plus package identity/MSIX or a sparse
package. The in-app installer is therefore a classic/Show-more-options bridge
until that native package layer is implemented and signed.

Generated actions:

- Extract Here
- Extract to Folder
- Compress to 7Z
- Test Archive
- Checksum

The task-window launch arguments are a shared GUI contract owned by
`crates/squallz-gui/src/open_files.rs`. Platform wrappers must inject those
argument names into their shell or PowerShell helpers instead of hardcoding a
second protocol.

Boundary:

- No Win11 `IExplorerCommand` COM handler yet.
- No MSI/NSIS/MSIX packaging integration for the Explorer verbs yet.
- No Windows code signing or SmartScreen validation yet.
- No live Windows VM smoke in the macOS-first phase.

## Linux

Generated assets include Nautilus scripts and a Dolphin service-menu file. They
are manual install assets because Linux desktops differ by file manager,
desktop environment, package format, and sandbox policy.

Generated actions:

- Nautilus scripts for Extract Here, Extract to Folder, Compress to 7Z, Test Archive
- Dolphin service menu with Extract Here, Extract to Folder, Test Archive
- Thunar command examples for manual custom actions

Boundary:

- No AppImage/deb/rpm installer yet.
- No xdg-mime/default-app claim yet.
- No Nautilus/Dolphin/Thunar live smoke in the macOS-first phase.
- WebKitGTK and distro-specific packaging checks remain required before Linux release.

## Generator

Platform integration assets are generated by the release packaging workflow.
The generated files are staged under `target/squallz-platform-integration/`
during packaging validation.

The generator validates generated file structure and shell syntax where the
current macOS host can do so. It does not install anything.
